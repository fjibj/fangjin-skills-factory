# Brain First — 用五步法激活自己的大脑，再问AI

> **Brain First. AI Second. Brain Last.**
> 人既是起点也是终点，AI 是中间的辅助。

---

## 为什么需要这个 Skill？

你有没有发现自己变成这样了：

- 遇到问题→打开ChatGPT/Claude→等答案→复制粘贴
- 自己的大脑好像"懒得动"了
- 写代码先问AI怎么写，而不是自己先想
- 久而久之，解决问题的能力不是提升了，而是**依赖感提升了**

这不是你的错。这是**"讨巧的顺序"**在作祟。

**事实是**：我们学会了给AI写很好的Prompt，但忘记给自己写Prompt了。

这个 Skill 就是来修正这个顺序的。它不是什么高深理论，就是**5个问自己大脑的题**，加一个**Final Test**。

## 安装

### 方式一：作为 Claude Code Skill 使用（推荐）

将本目录复制到你的项目：

```bash
# 复制 brain-first skill 到项目
cp -r brain-first/.claude/skills/brain-first.md your-project/.claude/skills/
```

然后在 Claude Code 中，输入：

```
/brain-first 帮我深度理解 [你要分析的主题]
```

### 方式二：作为独立工具使用（不需要Claude Code）

你不需要任何AI工具就能用这个方法。把下面的5个Prompt抄下来放在桌上，遇到任何复杂问题，**先走一遍再问AI**。

## 完整使用指南

### Step 1: "我已经知道什么？"

**什么时候用**：当你发现自己正要打开AI问问题的时候——停。

**怎么做**：拿纸笔或空白文档，写10分钟。

**问自己**：
```
关于 [主题]，我已经知道什么？
我大概猜什么是对的？
我能回忆起什么例子？
如果让我先试，我会怎么下手？
我具体卡在哪里？
我的第一反应是什么？
```

**完成标准**：你写出来的东西不需要漂亮。哪怕是一坨乱糟糟的笔记，只要是你自己写的，就行。

**然后做什么**：把你的笔记发给AI，告诉它："这是我的粗略想法，不要重写它，先告诉我——什么最强？什么不清楚？我漏了什么？怀疑论者会反对什么？哪个想法最值得深挖？"

---

### Step 2: "我假设了什么可能是错的？"

**什么时候用**：当你已经知道"大众观点"是什么的时候。

**背景**：大部分糟糕的思考不是因为蠢，而是因为戴着隐形假设。这个Prompt就是一把撬棍。

**问自己**：
```
关于这个主题，我假设了什么？（列出至少3个）
每个假设——如果它是错的呢？
如果相反的是对的，会改变什么？
我给自己发明了什么不必要的约束？
如果更简单，会是什么样子？
```

**完成标准**：你识别出至少一个你之前深信不疑的假设，并且认真推演了"如果反过来的话……"。

**真实案例**：当我们分析IDSD和Loop Engineering时，我一开始假设"IDSD是实践，Loop Engineering是理论"。但当我把这个假设反过来——"两者都是既有理论又有实践"——整个分析框架都变了。

**然后做什么**：把你的假设清单发给AI，让它帮你找更多你没意识到的隐形假设。

---

### Step 3: "给我十个坏主意"

**什么时候用**：当你发现自己陷入了"必须想出一个好主意"的焦虑中。

**为什么**：大脑有"完美过滤器"——它不允许你产生坏想法。但创造力需要数量在前，质量在后。第一个想法是显而易见的，第二个是借来的，第三个是想给人留下深刻印象的，第十个可能才真正有趣。

**问自己**：
```
围绕 [主题]，最荒谬/极端/故意走错/明显不对的想法是什么？
一个都不要筛选，只管写10个。
```

**完成标准**：你写满了10条。其中可能有些看起来是"好主意"——这说明你的潜意识在给你信号。

**真实案例**：在分析IDSD和Loop Engineering时，我写了"同一个项目分别跑两种流程比较结果"——这表面是个"坏主意"，实际上是最严谨的A/B测试框架。

**然后做什么**：把10个坏主意发给AI，让它帮你看——哪些坏主意里藏着金子。

---

### Step 4: "AI看到什么我没看到？我看到什么AI没看到？"

**什么时候用**：当你想真正搞清楚自己和AI的差异——不是为了比谁强，而是为了互补。

**为什么**：这一步是整个流程中学习真正发生的地方。你比较。你注意到AI看到了什么，你没看到什么。你也注意到你看到了什么，AI没看到什么。你发现AI在哪些方面更强，在哪些方面你那些"奇怪的小想法"更鲜活。

**问自己**：
```
第一部分：我看到了什么，AI（或一般讨论者）可能没看到？
- 我的独特经验、一手实践体感、踩过的坑
- 作为领域内的人，我知道的但外面不讨论的东西

第二部分：AI（或一般讨论者）看到了什么，我可能还没看到？
- 有哪些视角、数据、理论我还不够了解？
- 社区里大家在讨论什么我还不知道的？
```

**完成标准**：你写出了两列对比——"我有的"和"我缺的"。

**真实案例**：在分析IDSD时，我写了"意图编写其实也很花时间"、"编码和测试完全黑盒化"——这些都是我作为实践者才知道的。而AI反馈了"Loop的Evaluation可能有自欺成分"、"自然语言虽然不精确但场景化期望更可靠"——这些是我没深入考虑的方向。

**然后做什么**：发给AI，让它找出这两列之间的张力点——那个"矛盾又互补"的地方，往往藏着最深的洞察。

---

### Step 5: "如何把这个洞察锁进记忆？"

**什么时候用**：当你已经走了四步，脑子里有一堆想法但是散乱的。

**为什么**：偷来的想法很厉害，但也危险——它让你听起来比实际理解要聪明。目标不是听起来聪明，而是变得**更难被忽悠、更难被困惑、更难被替代**。

**问自己**：
```
穿过前面四步，我最重要的一个收获是什么？
用一句话写出来——用我自己的话，不要引用AI的话。

然后：
- 这个洞察为什么重要？对我的工作/判断/决策有什么实际影响？
- 我打算怎么记住它？（写便签？写进CLAUDE.md？发篇文章？讲给团队听？）
```

**完成标准**：你能用一句"人话"说清楚自己的核心洞察，并且有一个具体的"锁定记忆"行动。

**真实案例**：我的最终洞察是——"不管是IDSD还是Loop，自动化只是实现手段，人的作用才是重点。人既是起点也是终点，AI是辅助。"我打算写一篇公众号文章来记住它。

**然后做什么**：告诉AI你的洞察，让AI帮你打磨措辞（但不能替代你说话），然后帮你把洞察翻译成具体行动。

---

### Final Test: "不靠AI，你能讲清楚吗？"

**为什么**：如果离开AI你就说不清楚，那说明你还没有真正理解——你只是租借了流利。

**怎么做**：
```
关闭所有AI。不要复制粘贴之前的对话。

找一张纸或一个空白文档。向一位完全不了解这个主题的人解释：

"关于 [主题]，我最重要的理解是……"

条件：
- 不超过300字
- 用自己的语言、自己的比喻
- 不用AI帮你润色
- 写完之前不能回来
```

**通过标准**：
- ✅ 核心论点一句话能说清
- ✅ 用的是你自己的话（不是AI的措辞）
- ✅ 有"所以呢"——对方听完知道该怎么做
- ✅ AI不在场时也成立

**没通过怎么办**：没关系，回到Step 1重新走一遍。这次你知道更多了，你会注意到之前没看到的东西。

## 真实案例完整记录

为了让您更直观地理解这个Skill怎么用，以下是完整的实践记录。

### 背景
使用 Brain First 五步法 + Final Test 分析：
**"IDSD（Intent-Driven Software Development）和 Loop Engineering 到底是什么关系？"**

### Step 1 笔记（人写）
我写下了3个问题的答案：
- **Q1 IDSD**：人写意图（目标、约束、失败条件）和期望（场景化、Holdout Set），AI全自动设计/开发/测试
- **Q2 Loop Engineering**：Loop全自动，人只提需求和验证，/goal不达目的不罢休
- **Q3 关系？**：IDSD是实践，Loop Engineering是理论

### Step 2 假设挑战
我的假设是"IDSD是实践，Loop是理论"——但当我把这个假设反过来，发现两者都是既有理论又有实践的。我需要好好比较它们的具体实现。

### Step 3 十个坏主意
写了10个"坏主意"，其中最意外的发现：
- **#1** "两种流程各跑一次比较结果"——这不是坏主意，是最科学的A/B测试
- **#8** "holdout set就是用户故事，不用隐藏"——这暴露了一个致命误解：暴露holdout会让AI应试
- **#10** "意图由AI自动生成"——这击中了最危险的陷阱：让人继续偷懒

### Step 4 双向盲区对比
我看到的：
- 意图编写其实很花时间
- 编码和测试完全黑盒了
- 自然语言缺乏精确性

AI看到的：
- 都不能完全抛弃人的参与
- Loop的Evaluation可能有自欺成分
- 用户语言定义的期望其实更可靠

### Step 5 锁定洞察
**一句话**："不管是IDSD还是Loop，自动化只是实现手段，人的作用才是重点，人既是起点也是终点，AI是辅助。"
**怎么记住**：写一篇公众号文章。

### Final Test（不靠AI独立完成）
> *"经过 brain first 这几轮，我彻底了解了IDSD 和 loop engineering，其核心并不在于自动化，而是HITL(Human in the loop)，即人随着产品的迭代不断——不管是 BMAD 这种人工流程，还是 IDSD、loop engineering 这种自动化的流程——不断沉淀和提升自身能力，包括但不限于对产品的理解，能够准确的写出意图(目标、约束和失败条件)，定义期望(场景)或评估条件，在完善产品的同时完善自身，在提高效率的同时提升脑力。AI 是手段 ，人才是目标。这是我今天最大的收获。"*

**结论**：✅ Final Test 通过。

## 常见问题

### Q: 每次都要走完全部5个步骤吗？
**A**: 不一定。对于简单问题，走Step 1就够了。越复杂的问题，步骤走得越全。但**Step 1永远不能跳**——那是"反条件反射"的那一步。

### Q: 我没时间走完全部步骤怎么办？
**A**: 那就走Step 1 + Step 5。你至少先问了自己"我已经知道什么"，再问"我学到了什么"。这是最低配置。

### Q: AI Companion的角色到底是什么？
**A**: 把你自己的粗糙想法扔给AI时，指定它做这些事（不要让它直接写答案）：
1. 指出你的想法中什么最强
2. 指出什么不清楚
3. 指出你漏了什么
4. 模拟怀疑论者视角
5. 指出哪个想法最值得深挖

### Q: 这个方法和"先自己想想，再问AI"有什么区别？
**A**: 区别在**结构化**。大多数人的"先自己想想"就是发呆3分钟然后放弃。这5个Prompt是具体的、可执行的思考操作——每个都有问题、有完成标准、有下一步行动。

### Q: 可以用在什么地方？
**A**: 任何需要深度理解的场景：
- 技术选型（微服务 vs 单体）
- 论文阅读
- 系统架构设计
- 问题根因分析
- 团队技术决策
- 写文章/做演讲的资料准备

### Q: 用多了会怎样？
**A**: 你会发现两件事：
1. 你对AI的"答案饥渴"下降了——因为你发现你自己能想的比你以为的多
2. 你对AI回答的判断力提升了——因为有比较才有鉴别
